# 发布与打包方案

## 结论

本项目可以做成“用户直接可用”的发布版本，不必要求用户按开发流程手动安装前后端依赖。

当前代码默认是开发态运行，但可按以下三种方式发布：

1. 轻量发布（推荐先落地）
2. Windows 单文件可执行版（面向普通用户）
3. Docker 发布版（面向服务器）

---

## 方案一：轻量发布（推荐）

适用场景：内部团队、私有部署、希望发布快且稳定。

### 交付内容

- 后端 Python 代码（`src/`）
- 前端构建产物（`web-admin/dist`）
- 基础配置模板（`.env.example`）
- 启动脚本（如 `start.bat` / `start.ps1`）

### 用户侧要求

- 仅需安装 Python（无需 Node）

### 发布步骤（给维护者）

1. 在发布环境安装依赖（后端 + 前端依赖）。
2. 构建前端 `web-admin/dist`。
3. 将源码 + `dist` + 启动脚本打包成 zip。
4. 用户解压后执行启动脚本即可访问 `http://127.0.0.1:10010`。

### 优点

- 成本最低
- 与当前架构兼容最好
- 回滚和热修复简单

---

## 方案二：Windows 单文件可执行版（PyInstaller）

适用场景：非技术用户直接双击使用。

### 交付内容

- `bot.exe`（后端可执行程序）
- `web-admin/dist`（静态前端）
- `data/`（SQLite 数据目录）
- `README-使用说明.md`

### 用户侧要求

- 不需要 Python / Node

### 打包要点

1. 使用 PyInstaller 打包入口 `python -m bot.main` 对应模块。
2. 将 `web-admin/dist` 作为数据文件一并打包或旁路分发。
3. 确保 `DB_PATH` 使用相对路径（例如 `data/bot.db`）。
4. 首次运行仍走前端向导完成业务配置。

### 风险与注意

- 杀毒软件误报概率增加（需签名更稳妥）
- OpenAI/Telegram 依赖升级时要重新打包
- Windows 版本兼容性要在发布前验证

---

## 方案三：Docker 发布版

适用场景：云服务器或 NAS 持续运行。

### 交付内容

- `Dockerfile`
- `docker-compose.yml`
- 挂载卷（`/app/data`）保存 SQLite 与运行状态

### 用户侧要求

- 仅需 Docker / Docker Compose

### 运行方式

用户执行 `docker compose up -d` 后，直接访问 `http://<host>:10010` 完成首次向导。

### 优点

- 环境一致性最好
- 升级/回滚标准化
- 便于后续加监控和反向代理

---

## 推荐实施顺序

1. 先做方案一（最快上线可用版本）
2. 再做方案三（标准化部署）
3. 最后补方案二（面向桌面终端用户）

---

## 版本发布建议

### 版本号

- 使用语义化版本：`v主版本.次版本.修订号`
- 示例：`v0.2.0`（新增入群验证与 AI 欢迎语）

### 发布清单（每次发布前）

1. 后端单测通过（`pytest`）
2. 前端类型检查通过（`npx tsc -b`）
3. 前端构建产物存在（`web-admin/dist`）
4. `README` 与 `docs` 已同步更新
5. 生成发布说明（新增功能/兼容性/已知问题）

---

## 首个可用发布目标（建议）

- 发布类型：方案一（轻量发布）
- 目标：让用户“解压 -> 启动 -> 打开浏览器 -> 向导配置 -> 可用”
- 时间成本：最低，且基本不改当前架构
